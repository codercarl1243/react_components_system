# Building a Button Component

A Practical Implementation Guide

## Introduction

Buttons are the most reused interactive element in any interface — and the easiest to get wrong.

Most button implementations scatter responsibilities: styling logic lives inside the component, disabled state is handled inconsistently, async errors disappear silently. By the time a system grows, buttons have become a source of drift rather than a foundation of trust.

This post builds a button that does one job and does it predictably. It is type-safe, accessible by default, and backed by a reusable hook for interaction logic. It produces no visual output of its own — that responsibility belongs to the design system.

> **Context**
>
> This post uses the design system built across [Theming Foundations](/blog/design-system/theming) and [Primitives](/blog/design-system/primitives). Familiarity with those posts will help, but the button stands on its own.
>
> The implementation is deliberately minimal — the goal is a reliable, testable contract that teams can extend with confidence. Icons, spinners, and other additions should build on this foundation without breaking it.

## Component & Types

### Starting Simple

Before writing any code, it helps to be clear about what the button is responsible for:

- Managing synchronous and asynchronous click events
- Communicating loading and disabled states
- Preserving accessibility and focus behavior
- Preventing unintended interaction during inactive states

What the button is **not** responsible for:

- Deciding its own colors
- Defining its own spacing
- Applying visual variants

Those decisions belong to the design system layers established in the previous posts.

> **System rule**
>
> A component should own its behavior. The design system owns its appearance.

### The Component

The button composes from `Block` — the containment primitive introduced in the last post. By using `Block as="button"`, the component opts into the paint system without reimplementing it.

```tsx
'use client'

import clsx from 'clsx'
import type { ButtonProps, ButtonMouseEvent } from '@/components/button/button.type'
import useButton from '@/components/button/useButton'
import { Block } from '@/components/primitives'

export default function Button({
  className,
  children,
  onClick,
  onKeyDown,
  onKeyUp,
  type = 'button',
  disabled = false,
  isLoading = false,
  ...props
}: ButtonProps) {
  const { handleClick } = useButton()

  function onClickHandler(event: ButtonMouseEvent) {
    if (isLoading || disabled) {
      event.stopPropagation()
      event.preventDefault()
      return;
    }
    void handleClick(onClick)(event)
  }

  function onKeyDownHandler(event: React.KeyboardEvent<HTMLButtonElement>) {
    if (event.key === ' ') {
      event.currentTarget.dataset.pressed = 'true'
    }
    onKeyDown?.(event)
  }

  function onKeyUpHandler(event: React.KeyboardEvent<HTMLButtonElement>) {
    if (event.key === ' ') {
      delete event.currentTarget.dataset.pressed
    }
    onKeyUp?.(event)
  }

  return (
    <Block
      as="button"
      className={clsx(className, 'button')}
      aria-disabled={isLoading || disabled}
      type={type}
      data-testid="base-button"
      paint="all"
      {...props}
      data-loading={isLoading}
      onKeyDown={onKeyDownHandler}
      onKeyUp={onKeyUpHandler}
      onClick={onClickHandler}
    >
      {children}
    </Block>
  )
}
```

**Key decisions:**

- `Block as="button"` — the component inherits the paint system. `variant`, `variantAppearance`, and `paint` are available to consumers without any extra wiring. See [Primitives](/blog/design-system/primitives) for how `Block` works.
- `paint="all"` — the button always opts into full paint. Background, foreground, and border are all applied via the design system layer. See [Theming Foundations](/blog/design-system/theming) for how the paint layer is constructed.
- `aria-disabled` instead of `disabled` — keeps the button in the accessibility tree and maintains keyboard discoverability. More on this in the Accessibility section.
- `data-loading` — provides a CSS hook for cursor and interaction state styling without coupling to JavaScript.
- `data-pressed` via `onKeyDown`/`onKeyUp` — exposes keyboard press state as a data attribute so CSS can apply `:active`-equivalent styles when a user activates the button with Space.
- `type="button"` default — prevents accidental form submissions when multiple buttons exist in a form.

> **System rule**
>
> Critical behavioral props must not be overridable by consumers.

### TypeScript Support

```typescript
import type { MouseEvent } from 'react'
import type { BlockWrapperProps } from '@/components/primitives/types'

export type ButtonMouseEvent = MouseEvent<HTMLButtonElement>

export type ButtonClickHandler = (event: ButtonMouseEvent) => void | Promise<void>

type BaseButtonProps = {
  disabled?: boolean
  isLoading?: boolean
  onClick?: ButtonClickHandler
}

export type ButtonProps = Omit<BlockWrapperProps<'button', BaseButtonProps>, 'as'>
```

**Types breakdown:**

- `ButtonMouseEvent` — an alias that keeps the rest of the codebase clean.
- `ButtonClickHandler` — explicitly typed as returning `void | Promise<void>`, which communicates that we are not returning any value. This is consistent with React event handler patterns.
- `BaseButtonProps` — the button's own contract. Only the props the button itself owns: disabled state, loading state, and click handling.
- `BlockWrapperProps<'button', BaseButtonProps>` — inherits all native button props while merging the button's own contract. `variant`, `variantAppearance`, and `paint` come through from `Block` automatically. `as` is omitted because the button always renders as a `<button>` element.

## Interaction Logic

Every button interaction should feel complete — *"I acted. Something responded."*

Good buttons communicate. Users should always know their action was received, even if nothing has happened yet.

Behind the scenes, a custom hook handles interaction logic. The component stays thin while the hook ensures consistency across every button in the system.

The hook:

- Logs clicks in development
- Logs synchronous errors and re-throws them
- Attaches logging to unhandled Promise rejections
- Does not interfere with return values or error handling

```typescript
import type { ButtonClickHandler, ButtonMouseEvent } from '@/components/button/button.type'
import log from '@/lib/logging/log'

export default function useButton() {
  const handleClick = (userHandler?: ButtonClickHandler) =>
    (event: ButtonMouseEvent) => {
      if (!userHandler) return

      try {
        const result = userHandler(event)

        if (process.env.NODE_ENV !== 'production') {
          log()
        }

        /**
         * Note: this only catches rejections the user handler did NOT catch.
         * If the handler has its own try/catch, this won't run.
         */
        if (
          result !== null &&
          typeof result === 'object' &&
          typeof (result as PromiseLike<unknown>).then === 'function'
        ) {
          void Promise.resolve(result).catch((err) => {
            log()
          })
        }
      } catch (err) {
        log()
        throw err
      }
    }

  return { handleClick }
}
```

**Why these choices matter:**

- **Curried function** — `handleClick(onClick)(event)` lets the handler be configured once and reused predictably.
- **Duck typing for Promises** — checking for a `.then` method rather than `instanceof Promise` handles Promise-like objects correctly.
- **`void` operator** — signals intentionally that we do not await the Promise, preventing ESLint floating promise warnings.
- **Re-throw synchronous errors** — errors are logged, then propagate to React error boundaries as normal. The hook observes errors without suppressing them.

## Loading & States

![Three buttons showing default, loading, and disabled states](TODO: add image showing button states)

### Loading State

Buttons often trigger asynchronous actions. The system communicates this through `data-loading`, which CSS reads directly:

```css
&[data-loading="true"] {
  cursor: wait;
}
```

This keeps state communication out of JavaScript and in the layer that owns visual presentation.

### Preventing Duplicate Actions

When a button is loading or disabled, all interaction is blocked:

1. `isLoading || disabled` is checked before executing the handler
2. `event.preventDefault()` and `event.stopPropagation()` block the event entirely
3. `cursor: wait` or `cursor: not-allowed` communicates the state visually
4. `aria-disabled` communicates the state to assistive technology

### Form Integration

`type="button"` is set by default. This prevents unintentional form submissions and forces consumers to explicitly opt in to submit behavior:

```tsx
<Button type="submit" onClick={handleSubmit}>
  Save Changes
</Button>
```

## Accessibility

### WCAG Principles in Practice

The button addresses key accessibility requirements across all four WCAG principles:

- **Perceivable** — sufficient contrast (4.5:1) and readable text at 200% zoom. *(WCAG: 1.4.3, 1.4.11, 1.4.4)*
- **Operable** — fully keyboard accessible with clear focus states and appropriate target sizes. *(WCAG: 2.1.1, 2.4.7, 2.5.5)*
- **Understandable** — predictable behavior and clear state communication. *(WCAG: 3.3.2, 3.2.2)*
- **Robust** — proper semantic HTML and ARIA support for assistive technologies. *(WCAG: 4.1.2, 4.1.3)*

### The `aria-disabled` Decision

We use `aria-disabled` instead of the native `disabled` attribute.

`disabled` removes the element from the accessibility tree and tab order entirely. `aria-disabled` keeps the button focusable, ensuring it remains discoverable by keyboard users and screen readers.

We pair this with JavaScript click prevention — stopping propagation and default behavior to simulate a true disabled state while staying accessible.

> **System rule**
>
> Elements that are tabbable should remain tabbable regardless of their state.

> The button remains in the tab order when disabled. A user navigating by keyboard can still find it and understand that it exists but is currently unavailable. This is especially important for loading states, where the button will become active again shortly.

### Stable Sizing

![Button with minimum touch target dimensions highlighted](TODO: add image showing 44x44px target size)

Minimum dimensions enforce WCAG AAA compliance:

```css
min-width: 44px;
min-height: 44px;
```

Buttons that are too small are a problem not just for motor impairment — they are frustrating on touch devices for anyone, especially in motion (public transport, walking).

### Designing Beyond WCAG

WCAG gives us the floor, not the ceiling. A few considerations that go further:

- **Disabled contrast** — WCAG has an exception for disabled elements, but a disabled button should still be visually coherent. A completely invisible disabled button gives no signal that the action exists.
- **Stable layout** — the button's dimensions should not shift when its state changes. Layout shifts disorient users with cognitive disabilities and can cause accidental clicks on moving targets.
- **Predictable focus** — `cursor: pointer` on the base element, `cursor: not-allowed` when disabled, `cursor: wait` when loading. Each state communicates clearly without relying on color alone.

> **Note**
>
> Spacing between buttons is not enforced by the button itself — that responsibility belongs to the layout primitive. `Inline` or `Stack` controls the gap between interactive elements. See [Primitives](/blog/design-system/primitives) for how layout ownership works.

## CSS & Theming

The button's CSS owns structure and interaction affordances. It does not own color.

Color arrives through the design system layers: `variant` provides semantic tokens, `variantAppearance` maps those tokens, and `paint="all"` applies them. The button simply consumes the result.

### Reset & Base Styles

```css
.button {
  --button-font-size: 1rem;
  --button-padding-y: var(--spacing);
  --button-padding-x: var(--spacing-sm);

  /* WCAG Target Size: AA 24x24px minimum, AAA 44x44px minimum */
  min-width: 44px;
  min-height: 44px;
  white-space: nowrap;

  padding: var(--button-padding-y) var(--button-padding-x);

  -webkit-appearance: none;
  appearance: none;
  background: none;
  border-radius: var(--radius-md);
  outline: 0;

  box-shadow: var(--button-shadow, none);
  outline-offset: var(--button-outline-offset);

  font: inherit;
  font-size: var(--button-font-size);
  line-height: 1.25;
  text-align: center;
  vertical-align: middle;

  transition: all 0.2s ease;
}
```

### Interaction States

![Button showing hover, focus, active, and default states](TODO: add image showing button interaction states)

Interaction states use `--border-color` from the paint layer rather than hardcoded values. This means hover, focus, and active states adapt automatically to whatever variant the button is using.

```css
  &:hover {
    --button-shadow: 1px 1px 4px rgba(0, 0, 0, 0.2);

    outline: 1px solid var(--border-color, currentcolor);
  }

  &:focus-visible {
    --button-shadow: none;

    outline: 2px solid var(--border-color);
  }

  &:active,
  &[data-pressed="true"] {
    outline: 1px dotted var(--border-color, currentcolor);
    transform: translateY(1px);
  }
```

`data-pressed` mirrors `:active` for keyboard interactions. Space is the correct activation key for buttons per the ARIA button pattern — the browser fires the click on `keyup`, so `data-pressed` is set on `keydown` and cleared on `keyup`, giving the same visual pressed state that pointer users see from `:active`.

### Disabled & Loading States

```css
  &:is(:disabled, [aria-disabled="true"]) {
    --background-color: var(--color-neutral-200);
    --foreground-color: var(--color-neutral-800);
    --border-color: var(--color-neutral-400);

    cursor: not-allowed;

    &:hover,
    &:active {
      outline: none;
      box-shadow: none;
      transform: none;
    }

    &:focus-visible {
      --button-outline-color: var(--color-neutral-600);
    }
  }

  &[data-loading="true"] {
    cursor: wait;

    &:hover:not(:focus-visible) {
      outline: none;
      box-shadow: none;
      transform: none;
    }
  }
```

Disabled states override the paint layer's color tokens directly. This ensures disabled buttons remain visually coherent regardless of which variant is active.

### Ghost Appearance

![Ghost button showing hover and focus interaction feedback](TODO: add image showing ghost button states)

Most appearances (`filled`, `outlined`, `tonal`) are shared across the system and defined in the global appearance layer. Ghost is specific to interactive elements — it has no background by default but responds to interaction with subtle color mixing.

Because it is button-specific, it lives in a separate file:

**styles/design-system/appearances/button.css**

```css
.button[data-appearance="ghost"]:not(:disabled, [aria-disabled="true"]) {

  &:hover {
    --background-color: color-mix(
      in srgb,
      currentcolor 15%,
      var(--foreground-on-surface)
    );
  }

  &:focus-visible {
    --background-color: color-mix(
      in srgb,
      currentcolor 40%,
      var(--foreground-on-surface)
    );
    outline-offset: var(--spacing-sm);
  }

  &:active,
  &[data-pressed="true"] {
    --background-color: color-mix(
      in srgb,
      currentcolor 20%,
      var(--foreground-on-surface)
    );
    outline-style: dashed;
  }
}
```

Ghost uses `color-mix()` to derive hover and active backgrounds from `currentcolor` — meaning the interactive feedback is always relative to the button's foreground color, which is itself relative to the active variant. The appearance adapts correctly to any variant without any additional rules.

The `:not(:disabled, [aria-disabled="true"])` guard ensures ghost interaction styles do not apply when the button is inactive.

### Responsive & Motion

```css

@media (prefers-reduced-motion: reduce) {
  .button {
    transition: none !important;
    animation: none !important;
  }
}
```

Motion is disabled entirely when the user has indicated a preference — accessibility is applied at the system level, not as an afterthought.

### useButton Hook

The hook handles interaction logic independently of any UI. These tests validate it in isolation:

- **Event handling** — synchronous and asynchronous handlers run correctly
- **Error logging** — unhandled Promise rejections and synchronous errors are logged without interfering with execution
- **Return value** — the hook returns a stable curried function signature
- **Environment** — logging only runs outside production

### Button Component

The component tests focus on markup, accessibility, and behavioral correctness:

- **Rendering** — correct text, role, and prop application
- **Accessibility** — `aria-disabled`, keyboard focus, minimum touch target sizes
- **Loading state** — interaction is blocked, `data-loading` is set correctly
- **Interaction** — mouse and keyboard events behave predictably
- **Form integration** — `type="button"` default, submit override, disabled form prevention

## Summary

This post builds the base button as a thin behavioral layer on top of the design system. The component owns interaction logic and accessibility — nothing more.

- Structure comes from `Block`
- Color comes from `variant` and `variantAppearance`
- Paint is applied by the system, not the component
- Behavior is isolated in `useButton`
- Accessibility is built in, not bolted on

The result is a button that can be dropped into any context, given any variant, and trusted to behave correctly.

```tsx
describe('Button', () => {

    render(<Button>click me</Button>)
    expect(screen.getByTestId('base-button')).toHaveTextContent('click me')

    fireEvent.click(screen.getByTestId('base-button'))

    const handleClick = jest.fn()

  test('stops propagation when loading', () => {

      <div onClick={handleParentClick}>
        <Button isLoading onClick={jest.fn()}>click me</Button>
      </div>
    )
    fireEvent.click(screen.getByTestId('base-button'))
```

**useButton.test.tsx**

```tsx
import useButton from '@/components/button/useButton'
import { renderHook } from '@testing-library/react'
import { MouseEvent as ReactMouseEvent } from 'react'
import log from '@/lib/logging/log'
import withNodeEnv from '@/tests/helpers/withNodeEnv'

jest.mock('@/lib/logging/log.ts')

const mockLog = log as jest.MockedFunction<typeof log>

let handleClick: ReturnType<typeof useButton>['handleClick']

describe('useButton', () => {
  beforeEach(() => {
    jest.clearAllMocks()
    const { result } = renderHook(() => useButton())
    handleClick = result.current.handleClick
  })

  describe('handleClick', () => {
    test('accepts a curried function with the Event as the prop', () => {
      const mockUserHandler = jest.fn()
      const mockEvent = {} as ReactMouseEvent<HTMLButtonElement>
      handleClick(mockUserHandler)(mockEvent)
      expect(mockUserHandler).toHaveBeenCalledWith(mockEvent)
    })

    test('returns early when no handler is provided', () => {
      const clickHandler = handleClick(undefined)
      const result = clickHandler({} as ReactMouseEvent<HTMLButtonElement>)
      expect(result).toBeUndefined()
      expect(mockLog).not.toHaveBeenCalled()
    })

    test('does not return a value for sync functions', () => {
      const mockHandler = jest.fn()
      const clickHandler = handleClick(mockHandler)
      const result = clickHandler({} as ReactMouseEvent<HTMLButtonElement>)
      expect(mockHandler).toHaveBeenCalledTimes(1)
      expect(result).toBeUndefined()
    })

    test('logs and rethrows sync handler errors', () => {
      const error = new Error('Sync Error')
      const mockHandler = jest.fn().mockImplementation(() => { throw error })
      const clickHandler = handleClick(mockHandler)
      expect(() => {
        clickHandler({} as ReactMouseEvent<HTMLButtonElement>)
      }).toThrow(error)
      expect(mockLog).toHaveBeenCalled()
    })

    test('attaches logging to unhandled async errors', async () => {
      const error = new Error('Async Error')
      const mockHandler = jest.fn<Promise<void>, [ReactMouseEvent<HTMLButtonElement>]>()
        .mockRejectedValue(error)
      const clickHandler = handleClick(mockHandler)
      clickHandler({} as ReactMouseEvent<HTMLButtonElement>)
      await new Promise<void>(resolve => { global.setTimeout(resolve, 0) })
      expect(mockLog).toHaveBeenCalled()
    })

    test('does not log caught async errors', async () => {
      await withNodeEnv('production', async () => {
        const error = new Error('Caught Async Error')
        const mockHandler = jest.fn<Promise<void>, [ReactMouseEvent<HTMLButtonElement>]>()
          .mockImplementation(() => Promise.reject(error).catch(() => {}))
        const clickHandler = handleClick(mockHandler)
        clickHandler({} as ReactMouseEvent<HTMLButtonElement>)
        await new Promise<void>(resolve => { global.setTimeout(resolve, 0) })
        expect(mockLog).not.toHaveBeenCalled()
      })
    })

    test('logs clicks in non-production environments', () => {
      withNodeEnv('development', () => {
        const mockHandler = jest.fn()
        const clickHandler = handleClick(mockHandler)
        clickHandler({} as ReactMouseEvent<HTMLButtonElement>)
        expect(mockLog).toHaveBeenCalled()
      })
    })

    test('does not log clicks in production', () => {
      withNodeEnv('production', () => {
        const mockHandler = jest.fn()
        const clickHandler = handleClick(mockHandler)
        clickHandler({} as ReactMouseEvent<HTMLButtonElement>)
        expect(mockLog).not.toHaveBeenCalled()
      })
    })
  })
})
```
